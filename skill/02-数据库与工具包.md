# 数据库与工具包使用规范

## SQL 规则

- **使用引擎封装**：通过 `pkg/datastore/sqlstore` 引擎封装进行数据库操作，禁止在业务逻辑中散落原始查询。
- **条件意图显式**：每条查询必须明确表达筛选条件的业务意图，例如 `account = ?`。
- **错误映射边界**：在逻辑层边界将基础设施错误映射为领域错误（`pkg/err`）。
- **nil 实体作为业务状态处理**：查询结果为 nil 应视为业务上的「不存在」，而非基础设施故障。

## Redis 规则

- **Key 命名规范**：必须带命名空间且具有确定性，例如 `user:login:{user_id}:{platform}`；状态 key 与控制 key 分离，例如 `token:blacklist:{token}`。
- **nil 值语义**：`redis.Nil` 表示「键不存在」，不应作为致命错误处理。
- **TTL 对齐业务语义**：access token 过期时间、黑名单窗口期、nonce 生命周期须与业务约定一致。

## Mongo 规则

- **通过集中客户端访问**：使用 `svcCtx.Mongo.Collection(...).InsertOne/Find...` 进行操作，不得绕过注入的客户端。
- **集合/表名引用**：禁止硬编码集合或表名字符串，必须使用模型层定义的常量（如 `model.UserCollectionName`）。
- **审计写入不阻塞主流程**：登录日志等非关键审计写入不得阻塞主成功路径。
- **异步写入失败处理**：审计写入失败时，记录含标识字段的日志；除非有明确要求，否则不因此降级主流程的成功结果。

## 工具包使用规则

| 包 | 用途 |
|---|---|
| `pkg/err` | 所有对业务可见的错误统一使用类型化错误定义 |
| `pkg/page` | 分页列表 API 统一使用，避免自定义分页契约漂移 |
| `pkg/datastore` | 数据访问逻辑集中复用，禁止重复构建 repository |

## 事务与补偿策略

- **同一存储的多步操作**：使用严格事务（同一 datastore）。
- **跨存储的多步操作**：明确定义补偿策略（cross datastore）。
- **操作等级标注**：在编排代码附近的注释中注明哪一步是「尽力而为（best-effort）」、哪一步是「硬失败（hard-fail）」。
