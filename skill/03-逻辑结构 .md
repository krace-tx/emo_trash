# 逻辑分解与封装规范

## 核心原则

一个 logic 方法只完成一个业务用例；复杂分支提取为私有 helper 方法。

## 入口方法结构

logic 入口方法内推荐按以下顺序组织：

1. **入参规范化**：输入标准化与必要上下文提取
2. **实体查询**：加载目标用户/领域实体
3. **状态与权限检查**：业务约束校验
4. **核心操作执行**：主业务动作
5. **副作用处理**：缓存更新 / session 写入 / 审计日志 / 事件发布
6. **统一返回与错误映射**：构造响应或映射错误

## 提取为私有 Helper 的场景

以下重复逻辑序列须提取为独立的私有方法：

- token 生成（access / refresh）
- 平台标准化（platform normalization）
- 登录/会话 key 生成
- 审计日志写入
- 令牌黑名单检查与写入（isBlacklisted / addToBlacklist）
- 上下文身份提取（getUserIdFromCtx）

**Helper 编写约束**：

- 保持确定性，避免隐式依赖；所需值通过参数显式传入。
- 若 helper 失败不影响主流程，在代码注释中明确声明该策略。

## API vs RPC 逻辑职责边界

| 层 | 职责 |
|---|---|
| API logic | 请求适配 + RPC 调用 + 响应封包转换 |
| RPC logic | 业务策略 + 多存储编排 |

**禁止**：为了「简化 RPC」而将业务策略移入 API 层。

## 方法体量与可读性

- 入口方法尽量保持在一屏内可读。
- 若方法因多条件分支膨胀，按场景拆分为独立方法，例如：
  - `loginByPassword`
  - `loginByMobile`
  - `loginByThirdParty`
- 方法命名优先使用能表达场景的具体动词，避免 `handle`、`process` 等泛化命名。
