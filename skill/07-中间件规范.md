# 中间件定义规范

## 中间件职责范围

中间件只处理横切关注点，包括：

- 认证与 token 提取
- 授权与角色检查
- 请求追踪与日志 context 注入
- 限流与反滥用
- panic 恢复与统一输出映射

## 边界规则

- **禁止包含业务工作流决策**：如登录策略分支等逻辑属于 RPC 或专用 service logic。
- **重 datastore 依赖的领域检查**归属 RPC 层，不得迁移至中间件。
- 中间件应快速完成上下文丰富后将控制权传递给下游。

## 执行顺序建议

```
1. panic recovery（恢复）
2. request id / trace 注入
3. access 日志记录
4. 认证校验（auth verification）
5. 权限检查（permission checks）
6. 限流 / 风控（rate limit / risk controls）
```

## 认证中间件规则

- 从标准 header 或 cookie 位置解析 token。
- 使用集中认证工具函数验证签名与过期时间。
- 将用户身份信息注入 request context，供下游逻辑使用。
- token 无效时返回统一的未授权响应契约。

## 失败行为规则

- 中间件错误须返回确定性的状态码与 message。
- 拒绝路径上不得产生副作用（安全审计路径除外）。
- 审计日志写入失败不阻塞请求，合规强制路径除外。

## 新增中间件要求

新增中间件必须包含以下四项说明：

1. **单一职责声明**：明确本中间件只做什么
2. **顺序决策与理由**：在链中的位置及原因
3. **公开接口 bypass 策略**：哪些端点需要跳过此中间件
4. **基本测试用例**：覆盖成功路径与拒绝路径
