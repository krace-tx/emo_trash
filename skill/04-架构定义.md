# 架构定义规范

## 层级边界

| 层 | 路径 | 职责 |
|---|---|---|
| HTTP 传输层 | `app/api/gateway` | DTO 校验、请求路由、响应封包 |
| 业务域层 | `app/rpc/sso` | 认证/账号/会话的核心业务逻辑 |
| 基础设施层 | `pkg/*` | 可复用的基础组件与跨域工具 |

## 依赖方向

- **API 层** 可依赖 RPC 客户端与共享 `pkg`。
- **RPC 层** 可依赖 model、datastore 与 `pkg`。
- **`pkg` 层** 不得依赖任何 app 层模块。
- DTO 与领域模型须保持分离，禁止循环依赖。

## 契约规则

| 契约类型 | 规范 |
|---|---|
| 外部 HTTP 响应 | 统一使用 `CommonResp` |
| 内部 gRPC | proto 生成的 request/response 类型 |
| proto 字段变更 | 须同步更新：proto 定义 → 生成的 pb 文件 → gateway 请求映射 |

## 状态与 Session 归属

- **认证状态归属 SSO RPC 域**：token 黑名单、session 一致性策略不由 gateway 层持有。
- **中间件消费验证能力**：中间件调用认证工具函数验证 token，不得重写认证策略。

## 演进规则

- **新认证能力**：先在 RPC 层实现，再通过 API 层暴露，不得反向设计。
- **向后兼容**：保持现有 API 字段名与响应结构不变，除非有明确需求。
- **接口废弃**：须提供迁移路径与明确的错误行为定义。
