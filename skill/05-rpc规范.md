# RPC 实现规范

## 适用范围

`app/rpc/sso/internal/logic/**`、`server/**` 及所有 proto 驱动的认证流程。

## RPC 方法模板（标准执行顺序）

1. **实体查询**：从数据存储加载目标用户/实体
2. **状态与约束校验**：检查用户状态、业务规则约束
3. **核心操作执行**：凭证校验 / token 生成 / session 处理
4. **副作用持久化**：Redis 写入 / Mongo 日志 / 其他旁路存储
5. **返回响应**：返回 proto 类型响应，或返回 `pkg/err` 映射后的错误

## 错误处理规则

**预期失败场景**须使用 `pkg/err` 类型化错误返回，包括但不限于：

- 用户不存在
- 用户已禁用
- 密码不匹配
- token 生成失败

**基础设施错误**须 with context 记录日志，然后映射为领域级错误再返回。

**非关键副作用**（如登录日志写入失败）不得导致主请求失败。

## SSO 与 Token 规则

- `access_token` 与 `refresh_token` 须分别生成，使用各自独立的 secret 与过期时间。
- 同平台登录替换（SSO 踢出）：当策略开启时，新登录须使旧 token 失效。
- 旧 token 失效使用黑名单 key，须显式设置 TTL。
- 关键 session 状态的 Redis 写入失败须作为**硬失败**处理，不得静默忽略。

## 数据访问规则

- SQL 实体操作使用 `sqlstore.NewEngine[T]`。
- Redis 与 Mongo 通过注入的 `svcCtx` 访问，不得绕过服务上下文直接初始化客户端。
- 模型层假设须显式注释说明，例如用户状态枚举值的含义。

## 日志规则

- 失败与成功日志均须包含 `account`、`user_id`、`platform` 等标识字段。
- token 值须脱敏或完全省略。
- Info 级别日志仅记录关键业务里程碑，避免过度输出。
