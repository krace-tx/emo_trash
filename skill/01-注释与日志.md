# 注释与日志规范

## 注释规则

- **只注释有价值的内容**：业务意图、不变式、副作用、非显而易见的决策；禁止重复代码语义的注释。
- **语言风格一致**：与模块现有风格对齐；中文业务注释在本仓库中是可接受的。
- **长流程步骤注释**：对较长流程的关键步骤使用动词开头的短语，例如："检查用户状态"、"生成JWT令牌"、"写入登录日志"。
- **保留生成文件注释**：goctl 等工具生成的文件头注释不得修改。

## 日志规则

- **使用 context 绑定 logger**：使用 `logx.WithContext(ctx)`，日志位置应靠近失败点或关键分支。
- **日志级别区分**：
    - `Errorf`：操作失败
    - `Infof`：关键业务事件成功
    - 热路径中避免 debug 日志，减少性能噪音
- **稳定字段标识**：所有日志中保持一致使用以下字段（按可用情况携带）：`user_id`、`account`、`platform`、`device_type`、`device_id`、`ip`。
- **禁止记录敏感信息**：密码、token 完整值、短信/邮件验证码、加密盐、密钥一律不得出现在日志中。
- **日志格式模板**：`事件描述 + 执行结果 + 关键上下文`，例如："用户登录失败 account=xxx reason=密码错误"。

## 登录/Token 流程日志参考

| 场景 | 日志内容 |
|---|---|
| 查询失败 | 查询目标 + 标识符 |
| 校验失败 | 失败原因 + 操作者标识 |
| Token 生成失败 | 阶段（`access` 或 `refresh`） + 账号/用户 |
| SSO 踢出旧会话 | 旧 session 失效事件 |
| 登录成功 | 主体标识 + 平台，不记录 token 明文 |

## 错误映射规则

- **内部详情仅进日志**：返回给调用方的错误须映射为 `pkg/err` 中的类型化错误。
- **禁止泄露基础设施细节**：DB 错误、Redis 错误、堆栈信息不得出现在 API 响应中。
