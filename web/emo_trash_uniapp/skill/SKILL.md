# 移动端项目前端架构与 UI 设计规范文档

## 核心设计与工程理念 (Core Philosophy)
1. **内容即 UI (Content is UI)**：极简主义，摒弃多余边框与重色块，用大面积留白和文字排版划分层级。
2. **职责单一化 (Separation of Concerns)**：在代码架构上，严格分离“数据请求”、“业务逻辑”与“视图渲染”，禁止将所有逻辑揉捏在单一页面文件中。
3. **状态确定性 (State Determinism)**：任何 UI 的变化都必须由底层状态（State）驱动，保证交互反馈的及时性与防错性。

## 第一部分：UI 视觉设计规范 (UI Design Specifications)

### 1.1 色彩体系 (Color System)
保持高度克制，全站色彩分为三大类，主要通过灰度的变化建立视觉纵深。
* **品牌主题色 (Primary)**：用于最核心的操作（如：发布、确认登录、点赞激活）。需具备较高的饱和度与辨识度。
* **文字灰度 (Typography Colors)**：
    * **正黑 (#333333)**：用于大标题、核心正文，确保高对比度阅读。
    * **中灰 (#666666)**：用于次级标题、辅助信息、列表摘要。
    * **浅灰 (#999999/ #BDBDBD)**：用于输入框占位符(Placeholder)、时间戳、极弱提示。
* **背景与边框 (Background & Borders)**：
    * **卡片底色 (#FFFFFF)**：纯白，用于承载核心内容的卡片。
    * **页面底色 (#F7F8FA)**：带有极微弱暖意的浅灰，用于衬托白色卡片。
    * **分割线 (#EFEFEF)**：极浅，粗细限定为 `1px`（或 `0.5pt`），仅在必要时使用。

### 1.2 字体与排版 (Typography & Layout)
建立基于 `4pt/8pt` 的网格系统，所有间距、图标尺寸必须是 4 或 8 的倍数（如 8, 16, 24, 32）。
* **字号层级**：
    * 超大标题 (H1)：`44rpx - 48rpx`，字重 800，用于页面顶端无导航栏的沉浸式标题。
    * 常规标题 (H2)：`32rpx - 36rpx`，字重 500-600。
    * 正文 (Body)：`28rpx - 30rpx`，常规字重，行高设定为 `1.6`。
* **边缘安全区 (Safe Area)**：页面核心内容距离屏幕左右边缘统一保持 `32rpx` 间距。

### 1.3 核心组件库规范 (Components)
* **按钮 (Buttons)**：抛弃直角，采用全圆角（胶囊型）。主按钮高度统一为 `88rpx - 96rpx`。必须具备 `Default`（默认）、`Disabled`（不可用/置灰）、`Loading`（加载中）三种视觉状态。
* **输入框 (Inputs)**：去边框化。采用浅灰色圆角底板或极简下划线。聚焦 (Focus) 时必须有平滑的颜色加深过渡。
* **瀑布流卡片 (Masonry Cards)**：外圆角统一为 `12rpx`。图片必须预设占位图或骨架屏，防止图片加载导致的“叠罗汉”式排版崩塌。

## 第二部分：前端框架分层架构 (Framework Architecture)

为保障上述高标准的 UI 体验能够被稳定复现，前端代码实行**五层架构设计**，禁止越级调用。


### 2.1 视图层 (View Layer - `.vue` Pages/Components)
**职责**：只负责“长什么样”和“捕获用户动作”。
* **规范**：
    * 页面文件（Pages）代码行数控制在 300 行以内。
    * 杜绝在 `<template>` 中写复杂的 JS 逻辑（如连环的三元表达式）。
    * 视图层不直接调用 API，而是调用逻辑层提供的动作函数。

### 2.2 业务逻辑层 (Logic Layer - `composables/`)
**职责**：连接视图与数据，处理特定业务场景的流转（如“登录流程”、“列表分页拉取”）。
* **规范**：
    * 使用 Vue3 Composition API 将复杂逻辑抽离为 `useXXX.js`（例如 `useLogin.js`、`useWaterfall.js`）。
    * 在这里处理 UI 需要的衍生状态（如计算 `isLoading` 状态、控制骨架屏显隐）。

### 2.3 状态管理层 (State Layer - `stores/`)
**职责**：管理全局共享的业务数据。
* **规范**：
    * 采用 Pinia 进行状态管理，按业务模块拆分（如 `userStore`, `postStore`）。
    * 视图层跨页面通信（如在详情页点赞，回到列表页状态更新）必须通过 State 层，严禁使用混乱的全局事件总线 (EventBus)。

### 2.4 数据服务层 (Data Service Layer - `api/`)
**职责**：与后端服务器进行通信的唯一出口。
* **规范**：
    * 封装基础请求工具（如 Axios 或 uni.request），统一处理鉴权 Token、请求超时。
    * 统一拦截后端错误码（如 401 自动跳转登录页，500 弹出全局错误提示），绝不让错误对象直接抛到视图层导致 UI 崩溃。
    * API 接口按业务模块分文件（如 `api/auth.js`, `api/content.js`）。

### 2.5 基础设施层 (Infrastructure Layer - `utils/` & `constants/`)
**职责**：提供纯函数计算与全局配置。
* **规范**：
    * **Constants**：统一定义魔法字符串、枚举值、状态码映射表。
    * **Utils**：存放日期格式化、金额计算、本地缓存 (Storage) 封装等不依赖 UI 组件的纯函数。

## 第三部分：UI 与架构的协同机制 (Integration Rules)

设计与开发的最高默契，在于对**边缘状态（Edge Cases）**的统一处理规范。

### 3.1 异步交互反馈闭环
当视图层发起一个异步动作（如点击“登录”或“发布”）时，架构必须保障闭环：
1. **触发 (Trigger)**：用户点击按钮。
2. **锁定 (Lock)**：逻辑层立即将按钮状态设为 `Loading`，阻断用户的重复连击。
3. **通信 (Network)**：数据层发起 API 请求。键盘若处于弹起状态，需自动收起，防止遮挡提示。
4. **回显 (Feedback)**：
    * 成功：数据层返回数据，逻辑层更新 State，视图层触发跳转或弹出微弱的成功 Toast。
    * 失败：数据层拦截错误，视图层产生输入框抖动（Shake 动画）或在对应位置显示红色高亮提示语。
5. **重置 (Reset)**：解除 Loading 状态。

### 3.2 键盘与层级防御 (Z-Index & Safe Area)
* **防挤压策略**：页面滚动区域必须带有 `padding-bottom: env(safe-area-inset-bottom)` 属性。
* **全局层级表**：基础内容视图 `z-index: 1`；吸顶/吸底操作栏 `z-index: 100`；全屏弹窗/抽屉 `z-index: 1000`；系统级 Toast/Loading `z-index: 9999`。严格遵守，防止 UI 元素互相穿透。

---
**结语**
本规范旨在通过**极简的视觉语言**与**清晰的代码边界**，构建一个具备高可维护性与高用户体验的现代移动端应用。任何新模块的设计与开发，均需严格以此文档为准绳。
